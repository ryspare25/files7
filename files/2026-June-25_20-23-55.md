A reverse proxy is a server placed in front of backend servers.
Instead of the client contacting your application server directly:
Client → Application Server
the client contacts the reverse proxy first:
Client → Reverse Proxy → Backend/Application Server
The reverse proxy receives the client’s HTTP request, decides where it should go, forwards it to a backend server, receives the backend’s response, and returns that response to the client.
Why it is called “reverse”
A normal/forward proxy represents the client:
Client → Forward Proxy → Internet Server
For example, a company might use a forward proxy so employees access the internet through one controlled gateway.
A reverse proxy represents the server side:
Internet Client → Reverse Proxy → Internal Servers
The client may think it is communicating with one website, such as:
https://example.com
But behind that address, the reverse proxy may be distributing requests among many internal servers.

The backend servers can often remain private inside the company network. Only the reverse proxy needs to be publicly exposed.
Main uses
1. Load balancing
If one backend server receives too many requests, it can become slow or crash.
A reverse proxy can distribute requests:
Request 1 → Server A
Request 2 → Server B
Request 3 → Server C
Request 4 → Server A
This improves availability and prevents one machine from becoming overloaded.
Example:
Users
  |
Reverse Proxy
 /     |      \
App 1  App 2  App 3
If App 2 fails, the reverse proxy can stop sending traffic to it and send requests only to App 1 and App 3.
2. Hiding and protecting backend servers
Without a reverse proxy:
Internet → Backend Server
Your backend server’s IP address, ports, and infrastructure are more directly exposed.
With a reverse proxy:
Internet → Reverse Proxy → Private Backend Network
The backend servers can be placed in a private network where outside users cannot directly reach them.
This helps reduce exposure to:
Direct attacks against application servers
Port scanning
Unauthorized direct access
Some DDoS traffic
Attempts to exploit internal services
The reverse proxy becomes the controlled public entry point.
3. SSL/TLS termination
HTTPS encryption requires certificates and cryptographic processing.
Instead of configuring certificates on every backend server:
Client → HTTPS → App Server 1
Client → HTTPS → App Server 2
Client → HTTPS → App Server 3
the reverse proxy can handle HTTPS once:
Client → HTTPS → Reverse Proxy → HTTP/HTTPS → Backend Servers
This is called SSL termination or TLS termination.
The reverse proxy:
Receives the encrypted HTTPS request.
Decrypts it.
Sends the request to the appropriate backend server.
Receives the response.
Encrypts the response before returning it to the client.
This makes certificate management much easier.
4. Caching
A reverse proxy can save frequently requested content.
For example, if 100,000 users request the same image:
/logo.png
Without caching:
100,000 requests → Backend Server
With reverse-proxy caching:
First request → Backend Server
Later requests → Reverse Proxy Cache
This reduces backend work and makes the website faster.
Common cached content includes:
Images
CSS files
JavaScript files
Static HTML
Downloadable files
API responses, when appropriate
5. Routing requests to different services
A reverse proxy can route traffic based on the URL.
https://example.com/api/*      → API Server
https://example.com/images/*   → Image Server
https://example.com/admin/*    → Admin Server
https://example.com/shop/*     → Shop Server
Example:
Client
  |
Reverse Proxy
  ├── /api     → Spring Boot API
  ├── /web     → Vue/React frontend
  ├── /files   → File service
  └── /admin   → Admin application
This lets many separate applications appear as one website.
Real-world example
Suppose your company has three Spring Boot servers:
Spring Boot Server 1: 10.0.0.11:8080
Spring Boot Server 2: 10.0.0.12:8080
Spring Boot Server 3: 10.0.0.13:8080
Users do not access these directly.
Instead:
Users → https://mycompany.com → Nginx Reverse Proxy
Then Nginx forwards requests:
Nginx
  ├── → 10.0.0.11:8080
  ├── → 10.0.0.12:8080
  └── → 10.0.0.13:8080
To users, there is only one website:
https://mycompany.com
But internally, many servers are doing the work.

One-sentence definition
A reverse proxy is a server that receives client requests on behalf of backend servers, then forwards those requests to the appropriate internal server while providing features such as load balancing, caching, security, and SSL termination.
