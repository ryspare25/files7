1. Synchronous communication
In synchronous communication, the client sends a request and then waits until the server gives a response.
Client                     Server
  | ---- Request ------->    |
  |                          | processing...
  | <---- Response -------   |
  |
Client can continue only after receiving the response
Example: a user logs in.
User clicks “Log in”
        ↓
Browser sends username and password
        ↓
Server checks the credentials
        ↓
Server returns “success” or “invalid password”
        ↓
Browser proceeds
The browser cannot properly decide what to show next until it receives the result.
Why it is called “blocking”
The client is effectively blocked from completing that particular action while waiting.
That does not always mean the entire application freezes. A modern browser may still allow animations or other unrelated activity. But the specific workflow—such as submitting a form, loading a page, or confirming payment—depends on the response.
Typical examples
REST API request: GET /users/123
Login validation
Checking whether a product is in stock
Submitting a form and waiting for confirmation
Requesting account balance
Fetching a page from a server
Strengths
Easy to understand.
Easy to program for simple request-response actions.
The caller immediately knows whether the operation succeeded or failed.
Good when the next step truly depends on the answer.
Weaknesses
The user may wait a long time if the server is slow.
A server outage or timeout directly interrupts the user’s action.
If many requests wait on slow services, system resources can become tied up.
In distributed systems, waiting across multiple services can create cascading delays.
2. Asynchronous communication
In asynchronous communication, the client sends a request but does not wait for the final result. It can continue doing other work.
Client                     Server
  | ---- Request ------->    |
  |                          | processing continues...
  | ---- Does other work     |
  |                          |
  | <---- Result later ----  |
Example: uploading a large video.
User uploads a video
        ↓
System responds: “Upload received. Processing has started.”
        ↓
User can leave the page or continue using the app
        ↓
System processes the video in the background
        ↓
Later: “Your video is ready.”
The initial response may only confirm that the work was accepted, not that it was fully completed.
Important distinction: accepted vs completed
This is one of the most important ideas in asynchronous systems.
Synchronous:
“Your payment was completed.”

Asynchronous:
“Your payment request was received and is being processed.”
In asynchronous communication, the sender may receive an acknowledgment such as:
HTTP 202 Accepted
That means:
“The system received your request and intends to process it.”
It does not necessarily mean:
“The task has already finished successfully.”
Examples from the slide
WebSockets for real-time chat
A chat app cannot constantly make the user wait for manual refreshes.
Person A sends message
        ↓
Server receives it
        ↓
Server pushes it to Person B immediately
        ↓
Person B sees it in real time
This is often handled using WebSocket connections, where the server can send updates to the client whenever something happens.
Examples:
Chat applications
Multiplayer games
Live sports scores
Stock-price dashboards
Collaborative document editing
Delivery tracking
AJAX for background requests
AJAX allows a webpage to request data in the background without reloading the entire page.
For example:
User types “iph” in a search box
        ↓
Page sends a background request
        ↓
Server returns suggestions:
iPhone
iPhone case
iPhone charger
        ↓
Suggestions appear without refreshing the page
The user can keep typing while the request is happening.
Messaging applications
Messaging systems often use queues or brokers.
Order Service
      ↓
Message Queue
      ↓
Email Service
      ↓
Customer receives confirmation email
The order service does not need to wait until the email is sent. It can record the order first, then let another service handle the email later.
This is useful for:
Sending emails
Generating reports
Processing uploaded files
Creating invoices
Sending notifications
Updating analytics
Handling large batches of data
IoT applications
An Internet of Things device may send readings continuously:
Temperature sensor
      ↓
Message broker
      ↓
Monitoring system
      ↓
Alert system
The sensor should not wait for a long response every time it sends a temperature reading. It simply publishes data, and other systems react when needed.
Simple real-life analogy
Synchronous: ordering at a counter
You order food at a counter and stand there until the food is handed to you.
Order → wait → receive food
You cannot move on until the transaction is complete.
Asynchronous: getting a queue number
You order food, receive a queue number, and sit down or do something else.
Order → receive number → do other things → food is ready later
The queue number is like an acknowledgment or job ID.
Why this matters as systems grow
The sentence at the bottom of the slide means that the choice between synchronous and asynchronous communication affects the user experience and the architecture of the system.
A small system can often use synchronous calls everywhere:
Browser → Server → Database
But as a system becomes larger, it may have many services:
Browser
   ↓
API Gateway
   ↓
Order Service
   ↓
Payment Service
   ↓
Inventory Service
   ↓
Shipping Service
   ↓
Notification Service
If every service waits for every other service, one slow component can delay the whole chain.
Customer clicks “Place Order”
        ↓
Order waits for payment
        ↓
Payment waits for fraud check
        ↓
Fraud check waits for another service
        ↓
Customer waits...
This can create a poor experience.
An asynchronous design can separate urgent work from work that can happen later:
Customer clicks “Place Order”
        ↓
Order is recorded immediately
        ↓
Customer sees: “Order received”
        ↓
Background services process:
- confirmation email
- invoice
- warehouse notification
- analytics update
- delivery scheduling
The user gets a fast response, while non-critical tasks continue in the background.
When to use synchronous communication
Use synchronous communication when the caller needs the answer immediately.
Examples:
Is the password correct?
Is this payment approved?
What is the current account balance?
Does this user have permission?
Is this item available right now?
What data should this page display?
A useful rule:
If the next user action depends on the result now,
synchronous communication is usually appropriate.
When to use asynchronous communication
Use asynchronous communication when the task can happen later, may take a long time, or should not delay the user.
Examples:
Send an email
Resize an uploaded image
Generate a PDF report
Process a video
Notify warehouse staff
Update analytics
Run a backup
Send push notifications
Process thousands of records
A useful rule:
If the user does not need the final result immediately,
asynchronous communication is often better.
One practical example: online shopping
Synchronous parts
User logs in
User checks cart
User enters delivery address
User pays
The system needs immediate responses for these actions.
Asynchronous parts
Send receipt email
Notify warehouse
Update sales dashboard
Recommend related products
Prepare delivery tracking
These tasks can happen after the order is accepted.
So a real application commonly uses both approaches.
Synchronous for immediate decisions.
Asynchronous for long-running, background, or loosely connected work.
One-sentence summary
Synchronous communication makes the client wait for an immediate answer, while asynchronous communication lets the client continue working and receives the result or update later.
