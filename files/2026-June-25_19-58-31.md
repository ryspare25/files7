1. Explain the DNS resolution process step by step.
When a user enters a domain such as:
www.example.com
the device needs to discover the IP address associated with that name.
1. Browser checks its own DNS cache.
2. Operating system checks its DNS cache.
3. Device sends the query to a recursive DNS resolver.
4. Resolver checks its cache.
5. If not cached, resolver asks a root DNS server.
6. Root server points it to the appropriate TLD server, such as .com.
7. Resolver asks the .com TLD server.
8. TLD server points it to the authoritative DNS server for example.com.
9. Resolver asks the authoritative DNS server.
10. Authoritative server returns the final record, such as:
    www.example.com → 93.184.216.34
11. Resolver caches the answer for its TTL.
12. Resolver returns the IP address to the user’s device.
13. Browser connects to that IP address using HTTP or HTTPS.
A concise interview answer:
DNS resolution converts a domain name into an IP address. The client first checks local caches, then asks a recursive resolver. If the resolver has no cached answer, it queries the root server, then the TLD server, then the authoritative server. It returns the final IP address to the client and caches the result based on the record’s TTL.

4. What is TTL in DNS, and why is it important?
TTL means Time To Live.
It specifies how long a DNS record can remain cached before the resolver must request a fresh answer.
Example:
www.example.com → 93.184.216.34
TTL: 3600 seconds
This means DNS resolvers may reuse the cached answer for one hour.
High TTL
TTL: 86400 seconds
Equivalent to 24 hours.
Advantages:
fewer DNS queries
lower DNS-server load
faster repeat lookups
lower operating cost
Disadvantages:
changes take longer to propagate
failover may be slower
users may continue reaching an old IP address
Low TTL
TTL: 60 seconds
Advantages:
faster DNS changes
better for migration and failover
more flexibility during incidents
Disadvantages:
more DNS queries
more dependency on DNS availability
potentially higher resolver and authoritative-server load
A concise interview answer:
TTL defines how long a DNS record may be cached. A high TTL improves efficiency and reduces DNS traffic, while a low TTL allows faster changes and failover. The right TTL depends on the balance between performance, cost, and operational flexibility.
5. How does DNS-based load balancing work in large-scale systems?
DNS-based load balancing distributes users across multiple servers, regions, or endpoints by returning different DNS answers.
Example:
app.company.com
        ↓
DNS may return:
10.0.0.4
10.0.0.5
10.0.0.6
Different users may receive different IP addresses.
Common DNS load-balancing approaches
Method
How it works
Round robin DNS
DNS rotates through multiple IP addresses
Weighted routing
More traffic is sent to higher-capacity servers
Geographic routing
Users are directed to a nearby region
Latency-based routing
DNS chooses the endpoint with lower expected latency
Health-check routing
Unhealthy endpoints are removed from DNS responses
Failover routing
DNS sends users to a backup site if the primary site fails
Anycast DNS
Users reach the nearest available DNS node sharing the same IP

User in Philippines
        ↓
DNS resolves app.company.com
        ↓

        ↓
DNS resolves app.company.com
        ↓

user experience
Important limitation
DNS load balancing is not as immediate or precise as a Layer 7 load balancer because users and resolvers may keep DNS answers cached until TTL expires.
A concise interview answer:
DNS-based load balancing works by returning different IP addresses or endpoints for the same domain. It can use round robin, weighted, geographic, latency-based, health-check, or failover routing. It is highly scalable and useful for global traffic distribution, but DNS caching means traffic changes are not always instant.
