This slide shows how DNS is used inside a large-scale system, especially a cloud or enterprise network.
Diagram: what happens
A virtual machine requests the address for private.contoso.com from Azure DNS.
Azure DNS replies with the private IP address 10.0.0.4.
The VM then connects directly to the internal resource at 10.0.0.4.
Because this is a private DNS zone, the name resolves to an internal/private IP address rather than a public internet address. That allows systems in the same virtual network to communicate using readable names instead of memorizing IP addresses.
For example:
private.contoso.com → 10.0.0.4
Instead of an application being configured like this:
Connect to 10.0.0.4
it can be configured like this:
Connect to private.contoso.com
If the server later moves and its IP changes, administrators update the DNS record once; applications can continue using the same hostname.
Why DNS matters in large-scale systems
1. High availability
DNS can direct users toward different servers or endpoints.
If one server or region becomes unavailable, DNS can return another healthy server’s address. This helps reduce downtime.
Example:
app.example.com
   ├── Server A: 10.0.0.4
   ├── Server B: 10.0.0.5
   └── Server C: 10.0.0.6
DNS may distribute requests among them, or redirect traffic to a backup location when the primary environment fails.
This is why the slide mentions:
DNS load balancing
Anycast DNS
primary and secondary DNS servers
Anycast DNS means multiple DNS servers in different locations share the same IP address. A user is usually routed to the closest available DNS server, making DNS resolution faster and more resilient.
2. DNS failover strategies
DNS itself must not become a single point of failure.
A typical setup has:
Primary DNS server
        ↓ if unavailable
Secondary DNS server
If the primary DNS service fails, a secondary DNS server can still answer requests. In cloud systems, DNS zones are often replicated across multiple infrastructure locations automatically.
The important distinction is:
Application failover: redirecting users from a failed application server to a healthy one.
DNS failover: ensuring the DNS service itself can still resolve names even when one DNS server fails.
Both are necessary.
3. Content Delivery Networks and DNS
A Content Delivery Network, or CDN, stores copies of website content in multiple geographic locations.
DNS helps send a user to a nearby CDN edge server.
Example:
User in Manila
   ↓
DNS lookup for images.example.com
   ↓
DNS returns nearby CDN endpoint
   ↓
User downloads image from a closer server
Instead of every user downloading files from one server in another country, DNS can route them to a nearby copy. This reduces:
latency
page-load time
bandwidth pressure on the origin server
risk of one server being overloaded
4. Private DNS in cloud networks
The diagram is specifically about internal DNS resolution.
In a cloud environment, many services may have private IP addresses:
Database: 10.0.0.4
Application server: 10.0.0.8
Internal API: 10.0.0.12
Using private DNS, they can be reached through names:
database.private.contoso.com
api.private.contoso.com
app.private.contoso.com
This is better than embedding IP addresses in source code, configuration files, scripts, or deployment documents.
It also improves maintainability:
Without DNS
With DNS
Applications use hard-coded IP addresses
Applications use stable hostnames
IP changes can break connections
DNS record can be updated centrally

Harder to move resources between networks

5. DNS security risks
DNS is powerful because it tells systems where to connect. That also makes it a target.
The slide lists three common risks:
DNS poisoning
An attacker causes a DNS resolver to store a false record.
bank.example.com → attacker-controlled IP
A user may type the correct domain name but be directed to the wrong server.
Cache poisoning
This is a form of DNS poisoning where the false answer is stored in a DNS cache. Every later user who asks that DNS resolver may receive the malicious answer until the cache expires.
DDoS attacks
Attackers can overwhelm DNS servers with huge volumes of requests.
If DNS cannot respond, users may be unable to access the service even when the application servers are healthy.
Website server: healthy
Database: healthy
DNS: unavailable
Result: users cannot find the website
That is why large systems use redundant DNS providers, distributed DNS infrastructure, rate limiting, monitoring, and services such as Anycast DNS.
Core message of the slide
DNS is not merely “a phone book that converts names to IP addresses.”
In large-scale systems, DNS is part of the architecture for:
Availability
Performance
Failover
Load balancing
Private-network communication
Security
Global traffic routing

        ↓

At enterprise scale, that same basic mechanism becomes a major control point for where traffic goes, how quickly users reach services, whether systems survive outages, and how securely internal and public resources are accessed.
