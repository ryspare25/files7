The HTTP request–response cycle
This diagram shows what happens after a person enters a web address such as https://example.com into a browser.
The browser receives the URL
The browser sees the domain name, such as example.com. Humans use domain names because they are easier to remember than numeric IP addresses.
DNS translates the domain name into an IP address
The browser asks a DNS server:
“What IP address belongs to example.com?”
DNS responds with an address such as 93.184.216.34.
This is similar to looking up a business name in a directory to find its phone number.
The browser sends an HTTP request to the web server
Once it knows the IP address, the browser connects to the correct web server and sends a request, for example:
GET / HTTP/1.1
Host: example.com
GET means:
“Please give me the resource located here.”
The web server processes the request
The web server checks what the browser requested. It may:
return a static HTML file directly;
run application code;
check authentication;
query a database;
call another internal service;
generate HTML, JSON, images, or files.
In the diagram, the web server sends a query to a database, receives the requested data, and uses it to prepare the response.
The server sends an HTTP response
The server returns a response containing:
a status code;
headers;
the requested content.
Example:
HTTP/1.1 200 OK
Content-Type: text/html

<html>
  <body>
    <h1>Hello</h1>
  </body>
</html>
200 OK means the request succeeded.
The browser renders the webpage
The browser reads the HTML and displays the page. If the HTML refers to other resources—such as CSS, JavaScript, fonts, images, or API data—the browser makes additional requests for each of them.
Simple flow
User
  ↓ types URL
Browser
  ↓ asks for IP address
DNS Server
  ↓ returns IP address
Browser
  ↓ HTTP request
Web Server
  ↓ may query
Database
  ↑ returns data
Web Server
  ↓ HTTP response
Browser
  ↓ renders page
User sees webpage
Important distinction
The diagram simplifies a real system. In production, one “web server” may actually be many components working together:
Browser
  → DNS
  → CDN / Load Balancer
  → Web Server
  → Application Server
  → Cache
  → Database
  → Other APIs / Services
So when a user performs one action—opening a page, logging in, submitting a form, or clicking “Buy”—multiple infrastructure components may cooperate before the browser receives one response.
