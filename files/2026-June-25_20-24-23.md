Real-world examples of the client–server pattern
The slide is showing that the same basic idea appears in many systems:
A client asks for something; a server receives the request, processes it, and returns a response or pushes an update.
1. Web applications
Example: You open an online shopping site.
Browser → Web Server → Browser
Client: your browser, such as Chrome
Server: the website’s web server
Request: “Give me the homepage,” “show this product,” or “log me in”
Response: HTML, CSS, JavaScript, images, product data, etc.
Browser
   │  GET /products/123
   ▼
Web Server
   │  200 OK + product page/data
   ▼
Browser
The browser is the client because it initiates the request. The web server is the server because it waits for and handles requests.
2. APIs and microservices
Example: A mobile banking app checks your balance.
Mobile App → Backend API → Mobile App
Client: mobile app
Server: backend API
Request: “What is this user’s account balance?”
Response: account data, transaction history, error message, and so on.
Mobile App
   │  GET /api/accounts/balance
   ▼
Backend API
   │  validates user, retrieves data
   ▼
Mobile App
   │  displays ₱25,000 balance
With microservices, one server may itself become a client of another server:
Mobile App
   │
   ▼
API Gateway
   │
   ├──► Authentication Service
   ├──► Account Service
   ├──► Payment Service
   └──► Notification Service
For example, the payment service may call the notification service to send a receipt. In that interaction, the payment service is the client, while the notification service is the server.
3. Databases
Example: A backend application retrieves employee information.
Application Server → Database Server
Client: backend application, Java/Spring Boot application, SQL tool, etc.
Server: database server such as Oracle, MySQL, PostgreSQL, or SQL Server
Request: SQL query
Response: rows of data, confirmation of an update, or an error
SELECT employee_name, department
FROM employees
WHERE employee_id = 101;
Java Application
   │  SQL query
   ▼
Database Server
   │  matching employee record
   ▼
Java Application
The database server stores and manages data. The application does not usually read database files directly; it asks the database server to perform the operation.
4. Messaging systems
Example: A chat app sends and receives messages.
User A App → Messaging Server → User B App
Clients: the chat apps on User A’s and User B’s phones
Server: messaging server
Request/event: “Send this message”
Response/update: delivery confirmation, incoming message, read receipt, typing indicator
User A's Phone
   │  "Hello"
   ▼
Messaging Server
   │
   ▼
User B's Phone
For real-time communication, systems often use WebSockets rather than repeatedly asking the server, “Any new messages?”
Client ───────── persistent WebSocket connection ───────── Server
That lets the server push events immediately:
Server → Client: "New message received"
Server → Client: "User is typing..."
Server → Client: "Message was read"
This is common in chat systems such as WhatsApp and Slack.
The key point
The client–server pattern does not mean only “browser and website.”
It can describe:

Server

A system can also play both roles. For example, a backend API can be a server to a mobile app, but a client when it calls a payment API or database.
The core pattern remains the same: one component requests or consumes a service, while another component provides it.
