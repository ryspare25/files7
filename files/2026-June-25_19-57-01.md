Cache poisoning is when an attacker makes a DNS resolver save a false DNS answer in its cache.
Normally:
private.contoso.com → 10.0.0.4
A DNS resolver remembers that answer temporarily, according to its TTL—Time To Live—so it does not need to ask the authoritative DNS server every time.
With cache poisoning, the resolver is tricked into remembering something false:
private.contoso.com → attacker-controlled IP address
Then, even if users type the correct hostname, they may be directed to the wrong destination.
Simple flow
1. User asks DNS: “Where is bank.example.com?”
2. DNS resolver should return the real IP address.
3. Attacker injects or races a fake DNS response.
4. Resolver stores the fake response in its cache.
5. Later users receive the attacker’s IP address until the cache expires.
Example:
Correct record:
bank.example.com → 203.0.113.10

Poisoned cached record:
bank.example.com → 198.51.100.99
The user still sees and enters:
bank.example.com
But the browser is sent to the attacker’s server instead.
Why it is dangerous
Cache poisoning can enable:
fake login pages and credential theft
malware delivery
interception or redirection of traffic
access to a malicious server that impersonates a trusted service
disruption of legitimate services
It is especially dangerous because one poisoned DNS cache can affect many users, not only one victim.
How organizations reduce the risk
Use DNSSEC, which digitally signs DNS records so resolvers can verify authenticity.
Use unpredictable DNS transaction IDs and source ports.
Restrict recursive DNS resolution to trusted users or networks.
Keep DNS software patched.
Monitor DNS records and unexpected DNS-answer changes.
Use secure internal DNS zones for private resources.
Limit cache duration appropriately and investigate abnormal records.
Use HTTPS and certificate validation as an additional layer; HTTPS cannot prevent DNS redirection by itself, but a valid certificate makes convincing impersonation harder.
One-sentence summary
DNS cache poisoning is the corruption of a DNS server’s stored records so that legitimate domain names resolve to an attacker-controlled or incorrect IP address.
